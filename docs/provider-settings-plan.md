# LLM 프로바이더 설정 개선 계획 (2026-07-21 확정)

리뷰(UI/UX/프론트/백엔드/기획/운영자 관점) 결과, 근본 문제는 **설정이 IA에
자리가 없다는 것**(생성 작업 라우트 하단에 매몰)이고, 나머지 공백은 전부
"변경의 안전망"(검증·확인·기록·암호화)에 몰려 있다. 자리를 먼저 만들고
안전망을 채우는 순서로 간다.

## Phase 1 — "설정" 라우트 신설 (IA 근본 수정): **구현 완료 (2026-07-21)**

- 사이드바에 "설정" 항목 추가, 프로바이더 설정 카드 + 워커 카드를 이동.
  백엔드 API(`/api/settings/generation`)는 변경 없음 — UI 이동만.
- 생성 작업 화면에는 읽기 전용 요약 한 줄만 남긴다:
  "적용 중 — edit: … · t2i: … · 기획: … → 설정에서 변경" (상태 가시성 유지).
- 카드 제목의 개발자 흔적("— PUT /api/settings/generation") 제거.
- env 폴백 힌트를 placeholder 의존에서 상시 라벨/태그로 통일
  (키만 태그이고 모델·URL은 값 입력 시 힌트가 사라지는 비일관 해소).
- 스키마 변경 없음.

## Phase 2 — 변경 안전망 (피드백 루프 단축): **구현 완료 (2026-07-21)**

- **연결 테스트**: `POST /api/settings/generation/test` — 폼의 현재 입력값
  (미저장 포함)으로 fal 키 인증 확인 + LLM 엔드포인트 검증(경량 호출).
  UI에 이미지/LLM 테스트 버튼. 잘못된 키·오타 모델을 "다음 잡 실패"가
  아니라 저장 전에 발견하게 한다.
- **키 삭제 confirm** — 원클릭 null 전송에 확인 단계 추가.
- 모델·URL 필드에 "비우고 저장 = env로 복귀" 인라인 힌트 (같은 폼 안
  두 가지 빈값 의미론의 혼란 완화).
- DTO에 URL 형식 검증 추가.

## Phase 3 — 설정 변경 감사 로그: **구현 완료 (2026-07-21)**

구현 결정(사용자 지시로 원안에서 조정): 테이블 명은
**`console_logs`**(콘솔/관리자 행위)이며, 시스템 발생 이벤트용
**`service_logs`**(source/level/event_type/message/context_json)를 함께
신설했다. service_logs 최초 기록 지점은 durable 기록이 없던 두 곳 —
admin 워커의 게시 실패(DRAFT_PUBLISH_FAILED), service-backend의 DM 응답
실패(MESSAGE_REPLY_FAILED). 이미 CharacterActionLog가 커버하는 생성 잡
실패는 중복 기록하지 않는다.

- `CharacterActionLog`는 characterId FK 필수라 재사용 불가(확인됨) →
  **`admin_setting_changes` 테이블 신설** (스키마 오너 opod-service-backend):
  admin_id, key, 변경 유형(set/clear), 값 요약(키는 last4만, 원문 금지),
  created_at.
- PUT 핸들러에서 변경된 필드만 기록. 설정 화면에 최근 변경 이력 표시.
- 이력이 쌓이면 "이전 값으로 롤백"의 기반이 된다(롤백 UI 자체는 범위 외).

## Phase 4 — API 키 at-rest 암호화: **B안 결정 (2026-07-21, 사용자 확정)**

**결정: 리스크 수용.** admin_settings의 API 키는 평문 저장을 유지하고
at-rest 암호화는 도입하지 않는다. 방어선은 다음으로 한다:

1. 노출 API 전면 마스킹 — 키 원문은 어떤 응답에도 싣지 않음(`{set, last4}`).
2. DB 접근 통제에 위임 — DB에 닿을 수 있으면 어차피 서비스 전체가 뚫린
   수준이라는 판단.
3. 키 변경은 console_logs 감사 이력으로 추적.

**재검토 트리거**: DB 접근 주체가 늘어날 때(외부 분석 도구·제3자 연동),
침해 사고 발생 시, 컴플라이언스 요구 발생 시.

별건 권고(미이행): 예전 일회성 스크립트로 노출됐던 **운영 DB 비밀번호
로테이션**은 여전히 권장 상태다.

## Phase 5 — agent 채팅 LLM 설정 흡수 (기획 갭, 크로스 리포)

- 현재 캐릭터 채팅 LLM(opod-agent)은 env 전용이라 "LLM 관리"의 반쪽만
  admin에 있음. 실키를 admin DB에서 수동 복사해 쓰는 실태(E2E에서 확인).
- `admin_settings`에 `agent.*` 네임스페이스(chat 모델·키·임베딩 모델) 추가,
  admin 설정 화면에 섹션 신설, opod-agent가 DB 설정을 읽도록
  (채팅 핫패스이므로 TTL 캐시 재해석 — 잡마다 재해석하는 admin 워커 패턴의
  채팅판). 두 서비스 키 관리 채널 일원화.

## Phase 6 — 용도별 모델 분리: **보류 (관측 후)**

- 기획/프롬프트 빌드/캡셔닝이 LLM 설정 하나를 공유 — 용도별 오버라이드
  키(`planner.builderModel` 등)는 스키마 변경 없이 추가 가능하다.
- LLM 비용이 실제 문제로 관측되면 도입 (관측-후-전환 원칙).

## 범위 외

- 롤백 UI (Phase 3 이력이 선행 조건), 신규 프로바이더 플러그인 아키텍처,
  admin 권한 분리(설정 라우트 분리가 그 전제를 마련할 뿐).
